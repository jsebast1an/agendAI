## Objective

Build an AI-powered digital assistant for Teatro San Gabriel (Quito) that operates via WhatsApp and behaves like a real human receptionist.

The assistant must:

- Answer frequently asked questions (parking, prices, schedules, payment methods).
- Inform users about available performances by date.
- Check availability of functions.
- Allow ticket reservations.
- Escalate to a human when necessary.
- Maintain conversational memory (do not restart greetings if the same user sends another message).
- Support multiple concurrent users messaging at the same time.

This is not a simple chatbot.
It must behave like a professional digital receptionist - natural, helpful, and context-aware.

## Current Stack

- Backend: Laravel
- Frontend: React + Inertia
- UI: TailwindCSS
- Integration: WhatsApp Cloud API
- LLM: OpenAI API (ChatGPT)
- Database: MySQL

Current state:

- WhatsApp messages reach a Laravel webhook.
- The webhook sends messages to OpenAI and returns a response.
- Conversational memory is stored in the database (conversations + messages).
- The assistant loads only the last N messages into context (default 10) but keeps all messages for analytics.
- No tools / structured actions are implemented yet.
- Database schema exists for conversation memory (no plays/functions tables yet).
- No functions or ticket system implemented yet.

## New Product Direction (Manus-like Receptionist)

### What "Manus-like" means for AgendAI

We are not building a generic Q&A bot. We are building an **AI receptionist
system** where:

- The model is the conversational layer, not the source of truth.
- Laravel is the orchestration and business-rule engine.
- Domain tools/actions resolve real tasks (availability, pricing, schedule,
  reservation pre-flow, escalation).
- Every critical action is deterministic, auditable, and observable.
- The user experience remains human, warm, and concise.

### Target capabilities (next maturity level)

- **Intent understanding + slot filling** (date, play, quantity, contact data).
- **Tool/action execution** from backend (no SQL generated by LLM).
- **Structured memory layers**:
  - short-term conversational memory,
  - medium-term user profile/preferences,
  - long-term analytics/events.
- **Human handoff protocol** with explicit triggers and transcript continuity.
- **Operational reliability**:
  - retries/backoff,
  - idempotent inbound webhook processing,
  - traceable logs per conversation.
- **Prompt + policy governance**: versioned prompts and safe response rules.

## Data Architecture (Phase 1 Tools Foundation)

Current tables already in place:

- `conversations`
- `conversation_messages`

Next tables to implement:

- `venues`
  - Basic theater metadata (name, address, map_url, timezone, contact fields).
- `shows`
  - Show/play master data (title, slug, synopsis, duration_minutes, status).
- `functions`
  - Date/time instances for a show (show_id, venue_id, starts_at, status, capacity_total, capacity_blocked, sales_open_at, sales_close_at).
- `function_prices`
  - Prices by function and ticket type (function_id, ticket_type, currency, amount, is_active).
- `function_inventory` (or computed/materialized strategy)
  - Availability counters (function_id, seats_available, seats_reserved, seats_sold, updated_at).
- `faqs`
  - Structured FAQ source (topic, question, answer, locale, is_active, priority).
- `customers` (foundation for trusted profiles)
  - Identity and trust signals (phone_e164, full_name, trust_level, flags_json, last_seen_at).
- `reservations` (foundation for upcoming reservation phase)
  - Reservation lifecycle (conversation_id, customer_id, function_id, ticket_qty, status, expires_at, source).

Read-only tool contracts (first delivery):

- `get_showtimes(date)`
  - Input: ISO date in theater timezone.
  - Output: list of active functions with show title, start time, and status.
- `get_available_seats(function_id)`
  - Input: function identifier.
  - Output: seats available + sellability status.
- `get_ticket_prices(function_id)`
  - Input: function identifier.
  - Output: active price rows by ticket type and currency.
- `get_faq(topic)`
  - Input: normalized topic key.
  - Output: approved FAQ answers ordered by priority.

Backend validation rules:

- Never trust LLM text as source of truth for prices, schedules, or availability.
- All tool inputs are validated and normalized before querying.
- If data is missing, stale, or inconsistent, respond safely and trigger handoff.
- Log every tool call with conversation id, inputs, outputs summary, and latency.

Reservation policy (next phase, already aligned):

- Assistant can confirm reservation intent.
- Assistant can create reservation actions through backend.
- Pre-reservation is allowed only for trusted customers (`trust_level` policy + validations).
- Any ambiguity in identity, quantity, or function details forces clarification or handoff.

Handoff policy (mandatory triggers):

- Explicit customer anger/frustration.
- Complex request beyond defined tool capabilities.
- Repeated confusion after multiple failed lookups in same thread.

## Architecture Principles

- Keep the system simple and maintainable.
- The LLM interprets natural language.
- Laravel controls all business logic and database operations.
- The model must never generate raw SQL.
- Business rules live strictly in the backend.
- The assistant must feel human and professional.
- The system must be scalable for multiple simultaneous conversations.

### Additional principles for robust architecture

- **Backend-first truth:** all critical answers come from validated app data.
- **LLM as planner/interpreter, not executor:** model suggests, backend decides.
- **Explicit contracts:** tool inputs/outputs are typed and validated.
- **Observability by default:** every inbound/outbound step is traceable.
- **Graceful degradation:** if model/tool fails, respond safely and escalate.
- **Version everything:** prompts, routing rules, and policies.
